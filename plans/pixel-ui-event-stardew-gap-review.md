# Pixel UI 与事件体验深度审查

> 日期：2026-05-07
> 范围：Pixel mode 的界面、事件、运行中交互和玩家反馈。
> 参考标准：《星露谷物语》这类成熟像素游戏的第一分钟体验、事件表现、场景感和玩家反馈。
> 输入：`plans/pixel-game-issues.md`、Phase 13 review findings、当前 Pixel UI 代码。

## 结论

当前 Pixel mode 已经有可运行的像素前端，但它和成熟像素游戏的差距仍然很大。核心问题不是“缺几张图”这么简单，而是玩家看到的仍然是系统状态、节点、ID、效果列表和按钮面板；成熟像素游戏给玩家看到的是地点、角色、物件、危险、选择和结果。

更直接地说：现在它是一个有像素外观的交互原型，还不是一个像素 RPG。Phase 13 修掉了一批可读性问题，但界面和事件仍然缺少“场景先行、文字辅助、反馈可见”的基本游戏表达。

## 与《星露谷物语》的核心差距

### 1. 成熟像素游戏先给“地点”，当前游戏先给“界面”

《星露谷物语》里玩家进入任何区域，第一眼看到的是农场、房间、商店、矿洞、村庄、人物、可互动物。UI 只是在边缘辅助。

当前 Pixel mode 第一眼仍然是边框、按钮、状态条、节点格子和提示文字。地牢里虽然有格子和图标，但它更像路线图，不像一个房间或地点。

差距：

- 缺少明确地形：墙、地面、门、走廊、出口、入口。
- 缺少可互动物件：箱子、门锁、陷阱、机关、NPC、告示。
- 缺少环境层级：背景、前景、光源、障碍、路径。
- 文字占主导，场景承担的信息太少。

后续方向：

- Phase 14 必须把地牢从“节点图”改成“tile 场景”。
- Phase 17 再补标题场景、章节主题、Git 主题物件。

### 2. 成熟像素游戏用物件表达规则，当前游戏用文字解释规则

在成熟像素游戏里，宝箱打开后会变成打开状态，门锁会显示锁，危险物会有独立形状或动画，NPC 会面向玩家说话。玩家不需要先读一行规则才能知道发生了什么。

当前 Pixel mode 的奖励、陷阱、钥匙、宝库已经比 Phase 12 清楚一些，但关键反馈仍主要靠顶部小字和属性数字。

差距：

- 领取奖励后缺少稳定的打开状态和拾取动画。
- 陷阱只在触发后告诉损失，触发前预警不足。
- 宝库和钥匙已经有图形，但还没有形成一套完整图例和 hover/选中说明。
- 战斗胜利、事件结算、休息收益仍是文本结果，不是场景变化。

后续方向：

- Phase 14 增加已领取房间状态、陷阱预览、门/锁/宝箱 tile。
- Phase 18 增加视觉回归检查，避免这些状态后续被改坏。

### 3. 成熟像素游戏的事件是“小场景”，当前事件是“选择解析器”

《星露谷物语》的事件通常有地点、人物、动作、镜头节奏、对话和情绪。即使只是简单选择，也会先建立“发生了什么”，再让玩家选择“我要怎么回应”。

当前事件页拿到的是 `event_id`、`choice_id` 和 effects opcode。Pixel UI 展示的也是这些内部值。玩家看到的是“数据定义”，不是“冒险事件”。

当前事件链路的问题：

- `GameRunner.event_for_node()` 暴露给 Pixel UI 的内容只有事件 ID、选项 ID、效果 opcode。
- `EventScreen.draw()` 直接显示 `event_id` 和 `choice_id`。
- 选择后返回地牢时拼接 `Event: <choice_id> HP ... Gold ...`。
- 没有事件标题、描述、地点背景、人物、选择文本、后果预览。
- 没有事件发生前后的动画、音效层次、场景变化。

这导致事件的本质体验是：

> 玩家在一个面板里选一条数据，然后看到数值变化。

而成熟像素游戏事件应该是：

> 玩家进入一个地点，遇到一个具体状况，做出一个能理解的选择，并看到选择带来的后果。

后续方向：

- Phase 16 应该重做事件数据到 UI 的显示层。
- `EventSnapshot` 至少需要提供玩家标题、描述、选择文案、后果摘要。
- Pixel UI 不应该直接显示 `event_id`、`choice_id`、opcode。

### 4. 当前 UI 还没有统一“玩家语言”

Phase 13 已经把一部分 HUD 文案换成中文玩家语言，但还没有彻底完成。

仍然存在：

- 战斗胜利奖励里会出现 `EXP`、`Gold`。
- 技能不足时仍可能出现 `MP`。
- 事件结果仍会出现 `HP`、`Gold`。
- 事件页显示内部 ID。
- 商店英文原始字段在英文模式仍是 `cost hp atk mp maxhp`，中文模式虽有改善，但还不是商品文案。

成熟像素游戏不会把系统字段直接交给玩家。它会把同一含义统一成玩家词汇，比如“生命”“体力”“金币”“需要铁钥匙”“已打开”“无法通过”。

后续方向：

- 建立一个 Pixel UI 专用 player-facing formatter。
- 所有 screen 只拿 formatter 输出，不自己拼接 `HP/Gold/EXP`。
- 事件、战斗、商店、休息共用同一套属性词汇。

### 5. 暂停和退出语义仍不清楚

成熟游戏里“暂停”“退出本局”“返回标题”“关闭游戏”是不同动作，通常不会混在一起。

当前 PauseScreen 的问题是按钮写“Quit Run”，实际触发全局 quit，结果是关闭整个程序。这是严重的语义错误。

差距：

- 没有“返回标题页”的目标屏幕。
- 没有“确认退出本局但保留程序”的流程。
- 没有运行中设置入口。
- `Q` 暂停不一致，部分屏幕只响应 `Esc`。

后续方向：

- Phase 13 需要先修“退出本局不等于关闭程序”。
- Phase 16 再补完整暂停菜单：继续、设置、返回标题、退出程序。

### 6. 战斗表现仍像按钮面板

《星露谷物语》的战斗即使简单，也有空间位置、受击反馈、敌人动作、声音和掉落反馈。玩家看到的是角色和敌人在场景中发生互动。

当前战斗页能跑，但仍偏系统面板：

- 玩家和敌人是默认 sprite，没有动作区分。
- 攻击、格挡、技能反馈较薄。
- Boss 没有独立身份。
- 胜利后仍把奖励拼成一行文字。
- 操作按钮和日志挤在底部。

后续方向：

- Phase 15 应重做战斗分区：角色区、敌人区、动作区、日志区。
- Boss sprite、攻击动作、受击震动、掉落反馈要进入最低验收。

### 7. 商店和休息点缺少“角色/地点”

成熟像素游戏里的商店不是商品列表，而是一个地点：有店主、柜台、货架、价格、物品图标和购买反馈。休息点也不是选项面板，而是营火、床、温泉、神龛或安全房。

当前：

- 商店仍以列表、按钮、数值为主。
- 商品缺少图标、名字、价格牌、买不起原因的视觉状态。
- 休息点只是两个选项，没有营地或安全地点的场景感。

后续方向：

- Phase 16 把商店、休息、事件从“数据面板”改成“场景 + 选择”。
- 不要先加更多商品，先让一个商品和一个休息选择看起来像游戏。

### 8. 测试还不能证明“像游戏”

现有测试能证明逻辑、输入、parity 和 smoke，但不能证明玩家看到的画面可读。

缺口：

- 没有检查中文截图是否还混英文属性词。
- 没有检查标题、地牢、事件页是否文字重叠。
- 没有检查暂停页按钮文案和行为是否一致。
- 没有检查事件页是否出现内部 ID。
- 没有检查奖励、陷阱、钥匙、宝库是否有独立视觉状态。

后续方向：

- Phase 18 必须加入 screenshot 或像素边界检查。
- Phase 16 事件重做时要加“禁止显示内部 ID”的测试。

## 事件系统应达到的最低标准

在进入“像素游戏事件”之前，事件系统至少要满足：

- 玩家看到事件标题，而不是 `event_id`。
- 玩家看到事件描述，而不是效果 opcode。
- 每个选项有自然语言文案，而不是 `choice_id`。
- 每个选项显示可理解的后果预览，比如“恢复生命”“失去金币”“获得钥匙”。
- 选择后有明确反馈，不出现 `HP`、`Gold` 这类混杂词。
- 事件页有地点或角色视觉元素，不只是一个面板。
- 事件完成后回到地牢时，地牢提示能用玩家语言总结结果。

## 界面系统应达到的最低标准

在进入“完整像素 UI”之前，界面系统至少要满足：

- 任意运行中屏幕按 `Esc/Q` 的行为一致。
- “退出本局”“返回标题”“关闭游戏”三者不混用。
- 普通玩家界面不显示内部 ID、slot、opcode、原始字段名。
- 中文模式没有半截英文属性词。
- 每个 screen 有固定布局区：标题、状态、主场景、动作、提示。
- 鼠标能点击看起来可点击的东西；不能点击时有解释。
- 重要结果有画面变化，不只是一行文字。

## 推荐修复顺序

### 立即修复：Phase 13 补丁

1. 把 PauseScreen 的“退出本局”改成返回标题或改文案为“关闭游戏”。推荐前者。
2. 战斗胜利奖励使用统一 formatter，不再拼 `EXP/Gold`。
3. 事件、休息、商店都支持 `Esc/Q` 进入暂停。
4. README 当前文档索引更新到 Phase 13 完成、Phase 14 待开始。

### 下一阶段：Phase 14

1. 把地牢节点图改成 tile 房间。
2. 加门、墙、走廊、宝箱打开状态。
3. 陷阱靠近或选中时显示风险。

### 后续重点：Phase 16

1. 重做事件数据暴露层，禁止 UI 直接拿 ID/opcode。
2. 给事件页增加标题、描述、选择文案和后果预览。
3. 商店和休息点改成场景。

### 验收前：Phase 18

1. 加中英文截图检查。
2. 加禁止内部 ID 出现在玩家界面的测试。
3. 加暂停语义测试。

## 最终判断

如果只和 CLI 版相比，Pixel mode 已经是明显进步；如果和《星露谷物语》这类成熟像素游戏相比，它还缺三个关键层：

- **场景层**：玩家看到的是地点，而不是节点和面板。
- **叙事层**：事件有角色、地点和原因，而不是 ID 和效果。
- **反馈层**：动作结果进入画面，而不是只进入文字和数字。

这三个层不补齐，继续加房间、奖励或事件数量，只会让原型变复杂，不会让它更像一个真正的像素游戏。
